# Path of Code

A community-driven series of programming challenges focused on learning, practice, and personal growth.

Check out our discord server: [Alex's Disciples](https://discord.gg/XaduDKeBt)

---

## What is Path of Code?

Path of Code is a collection of programming challenges designed to help developers improve their problem-solving skills, programming fundamentals, and software engineering knowledge through hands-on practice.

Each challenge is carefully designed around one or more educational objectives. While participants will be given a specific task to complete, every challenge also contains underlying concepts intended to strengthen their understanding of computer science and programming.

Topics may include:

* Data Structures
* Algorithms
* File Systems
* Networking
* Web Servers
* Memory Management
* Object-Oriented Programming
* Functional Programming
* Concurrency
* Software Architecture
* Design Patterns
* And many others

Whether you are a beginner writing your first data structure or an experienced developer exploring unfamiliar topics, Path of Code aims to provide meaningful opportunities for growth.

---

# Philosophy

> Worse than tomorrow, better than yesterday.

Many modern tools can generate complete solutions instantly. While those tools can be useful in many situations, they often remove the most valuable part of the learning process: struggling with a problem, experimenting, making mistakes, and discovering solutions independently.

For this reason, participants are encouraged to solve challenges using their own knowledge and reasoning.

Path of Code is built around the belief that genuine understanding comes from practice.

# Rules

## Allowed Resources

The following references are permitted:

* [LEARN C++](https://learncpp.com/)
* [cppreference](https://cppreference.com/)

Participants may also:

* Ask questions to moderators
* Use the community help channels (**#ask-for-help**)
* Discuss concepts without sharing complete solutions

---

## Not Allowed

To preserve the educational value of the challenges, the following resources should not be used:

* AI assistants (ChatGPT, Claude, Gemini, Copilot, etc.)
* Search engines for solutions
* Stack Overflow
* Reddit
* Tutorials that directly explain the challenge solution
* Public repositories containing completed implementations

The purpose of these restrictions is not to make challenges harder, but to maximize learning.

---

# Challenge Structure

Each challenge contains:

* Problem description
* Objectives
* Requirements
* Submission instructions
* Scoring information
* Optional bonus objectives

Challenges are designed to gradually increase in complexity.

A participant should be able to complete early challenges with basic programming knowledge while more advanced participants can explore bonus objectives and advanced implementations.

# How to Participate

## Step 1 - Fork the Repository

Click the **Fork** button at the top of this repository.

This creates your own copy of the repository under your GitHub account.

---

## Step 2 - Clone Your Fork

Clone your fork locally:

```bash
git clone https://github.com/YOUR_USERNAME/Path-of-Code.git
```

---

## Step 3 - Complete the Challenge

Create your solution inside your fork.

Participants are free to organize their projects however they prefer, provided that:

* The solution can be compiled successfully.
* The solution can be executed successfully.
* The project contains proper documentation.

Preferably put your solution inside the specific challenge directory:

```plain
Path-of-Code/
│
├── Challenges/
│   ├── challenge-001-sorting_algorithms/
│   │   ├── Readme.md << (add your instructions here)
│   │   ├── Leaderboard.md
│   │   ├── header/
│   │   │   └── sort.h
│   │   │
│   │   ├── main.cpp << (not required, but recommended)
│   │   └── sort.cpp
│   │
│   ├── challenge-002-linkedlist/
│   ├── challenge-003-.../
│   └── ...
│
├── Leaderboard.md
└── Readme.md
```

---

## Step 4 - Write Documentation

Every submission must include a README.md file containing:

* Project description
* Compilation instructions
* Execution instructions
* Implemented features
* Notes about the solution

Participants are strongly encouraged to explain:

* Their thought process
* Difficulties encountered
* Lessons learned
* Design decisions

Understanding your own solution is often more valuable than the solution itself.

---

## Step 5 - Create a Pull Request

After finishing your solution:

```bash
git add .
git commit -m "Complete Challenge XX"
git push
```

Then:

1. Open your fork on GitHub.
2. Select **Contribute**.
3. Select **Open Pull Request**.
4. Submit your solution.

---

# Recommended Project Practices

Although not mandatory, participants are encouraged to:

* Use meaningful variable names
* Write readable code
* Keep functions focused on a single responsibility
* Separate implementation into multiple files when appropriate
* Use source control properly
* Document important decisions

A `.gitignore` file is also recommended to avoid committing:

* Build artifacts
* Temporary files
* IDE configuration files
* Executables and binaries

Repositories should ideally contain only source code and documentation.

---

# Scoring

Each challenge may define its own scoring system.

Typically points can be earned through:

* Completing required objectives
* Completing bonus objectives
* Submitting before the deadline
* Demonstrating understanding through documentation

Specific scoring rules will be defined inside each challenge.

---

# Getting Help

Learning is a collaborative process.

If you become stuck:

* Ask questions in the designated help channels
* Reach out to moderators
* Discuss ideas and concepts with other participants

However, avoid requesting or sharing complete solutions.

The objective is to help each other learn, not to bypass the challenge.
